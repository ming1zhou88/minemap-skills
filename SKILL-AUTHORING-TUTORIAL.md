# MineMap Skill 生成教程：方法、经验与落地流程

这份文档不是介绍某一个 MineMap API，而是总结这套 `minemap-skills` 是怎么被建立出来的。

目标有两个：

1. 教你如何继续写新的 skill
2. 分享一套已经在 MineMap 4.22 源码上验证过的整理方法，尽量让结论正确、准确、好用

## 先说一段口语化经验

如果只用一句大白话来总结，这套 skill 的写法其实就是：

**别急着写，先把源码看明白；别光抄 demo，先分清哪些能正式用；别为了写全而乱堆，优先把真正常用、容易写错的地方讲清楚。**

再说直白一点，就是下面这几条：

-   先看源码，不然很容易把过时写法也当成推荐写法
-   demo 可以参考，但不能照单全收，很多只是演示方便
-   一个 skill 不用啥都包进去，能把一类问题讲透就已经很有用了
-   最有价值的不是“参数大全”，而是“遇到这个需求到底该从哪里下手”
-   写 skill 本质上不是写说明书，是帮后面的人少踩坑、少走弯路

所以实际干活的时候，我一般就是按这个顺序来：先找公开入口，再看实现，再看 demo 怎么组织，最后才动手写。这样虽然慢一点，但写出来的东西更稳，也更能直接拿去用。

## 1. 先明确：我们在写的不是“普通说明文档”

这套 skill 的目标不是百科式介绍，也不是把官方文档重抄一遍。

它的真正目标是：

-   给智能编码助手提供更可靠的上下文
-   给开发者提供更接近真实工程的 API 入口判断
-   把“源码存在但不好找”“demo 有示例但边界不清楚”的知识整理成可直接使用的规范

所以 skill 文档必须回答的是下面三类问题：

1. 这个能力到底应该从哪个公开入口进入？
2. 这个能力有哪些真实约束、时序要求、失败点？
3. 哪些 demo 写法可以复用，哪些只是 demo 调试写法，不能当正式方案？

如果一份文档只是概念解释，却不能帮助写代码，那它就不是一份合格的 skill。

## 2. 整体生成逻辑：先核源码，再核 demo，最后入库

整个 skill 生成逻辑可以概括成一句话：

**先核源码，再核文档注释，再核 demo，最后才把结论写进技能库。**

推荐优先级如下：

### 第一层：源码是真实边界

优先看这些目录：

-   `source/`
-   `source/api/`
-   `source/index.js`
-   `source/style/`
-   `source/api/control/`
-   `source/renderer/`
-   `source/webgl/`

为什么源码优先：

-   它决定真实导出入口
-   它决定参数是否仍然有效
-   它能看出哪些路径已经废弃、哪些只是兼容保留
-   它能识别对象真实归属，例如是 `Map` 方法、scene component，还是低层对象

举例：

-   三维加载的推荐入口，必须先看 `source/api/map.js`
-   某个类是否公开导出，必须先看 `source/index.js`
-   某个控件是否真的属于 `IControl`，必须先看 `source/api/control/*.js` 和 `source/util/doc-use/doc-icontrol.js`

### 第二层：`docs/src-cn/` 是官方注释源，不是唯一真相

`docs/src-cn/` 很有价值，因为它提供：

-   参数注释
-   类型说明
-   历史迁移提示
-   公开 API 的表述口径

但它不能单独当真相来源。

原因很简单：

-   注释可能滞后于实现
-   某些 demo 或实际实现已经转向新入口
-   个别旧路径虽然文档还在，但源码已经不推荐继续扩散

所以正确做法是：

-   先从 `docs/src-cn/` 找说明
-   再回到 `source/` 核实这个说明在当前版本是否仍然成立

### 第三层：`demo/html/` 用来验证“怎么组织一套可跑流程”

demo 的价值不在于“抄代码”，而在于它能告诉你：

-   一个能力通常挂在哪个事件阶段执行
-   多个对象如何组合
-   参数怎样组织更稳定
-   某些功能在工程里是如何被串起来的

比如：

-   `load` 后初始化还是 `style.load` 后初始化
-   一个工具控件是否需要和 GUI、scene component、事件联动一起使用
-   查询交互功能是以图层事件组织，还是以 GPU pick 组织

但 demo 也有风险：

-   会出现调试代码
-   会出现内部字段写法
-   会出现临时页面逻辑，不适合进入正式 skill

所以 demo 的作用是“验证流程”，不是“自动升格为正式 API”。

## 3. 一份新 skill 的标准产出流程

下面是一套建议固定执行的步骤。

## Step 1：先确定专题边界，而不是先写内容

不要一上来就写文档。

先回答：这份 skill 到底解决哪一类问题？

例如：

-   `minemap-style-system` 解决的是 style 结构与 `setStyle()` 工作流
-   `minemap-layer-system` 解决的是图层增删改移、顺序、过滤、缩放范围
-   `minemap-widget-and-controls` 解决的是 `IControl`、`addControl()` 生命周期和工具型控件边界

一个 skill 最怕两种情况：

1. 过大：把几个完全不同的主题揉到一起
2. 过碎：一个非常小的 API 单独拆一包，维护成本过高

经验上，可以用这个判断法：

-   如果这些能力共享同一入口和生命周期，适合放一起
-   如果它们的对象模型、时序、失败点明显不同，就应该拆开

## Step 2：从 `source/index.js` 和核心入口找“公开边界”

新专题一旦确定，先做公开入口盘点。

建议优先查：

-   `source/index.js`
-   `source/api/map.js`
-   对应类所在目录，例如 `source/api/control/`、`source/style/`、`source/renderer/effects/`

要回答的问题包括：

-   这个能力是否真的公开导出？
-   真实入口是 `Map` 方法、构造类，还是静态工具？
-   有没有推荐的新路径？
-   有没有旧路径还在，但其实不该继续推广？

这一步完成后，skill 的“主语”才会准。

## Step 3：从 `docs/src-cn/` 补参数语义与迁移线索

源码解决“有没有、在哪儿”，文档注释解决“参数是什么意思、官方想怎么描述”。

这一轮重点看：

-   构造参数
-   事件名
-   返回值
-   废弃提示
-   兼容迁移关系

这一步的产出不是直接写文档，而是形成一个参数与术语清单。

## Step 4：从 `demo/html/` 找真实可跑模式

这一步主要做两件事：

1. 验证前面的结论是否能跑通
2. 提炼“推荐组织方式”

推荐观察这些东西：

-   何时初始化
-   何时 add/remove
-   事件如何绑定
-   对象之间如何联动
-   结果如何导出、回写、调试

如果一个能力在多个 demo 中反复出现，就说明它已经形成稳定模式，应该优先写入 skill。

## Step 5：过滤掉三类不该进入 skill 的内容

### 不该进 skill 的内容 1：私有字段

只要写法依赖：

-   `_privateField`
-   `_internalState`
-   没有公开导出的内部对象

都不要写成推荐方案。

### 不该进 skill 的内容 2：历史路径扩散

如果源码已经明确有新入口，就不要继续扩散旧路径。

典型例子就是三维数据：

-   新代码优先 `map.addSceneComponent()`
-   不继续把旧 `3d-tiles` source/layer 路径写成推荐方案

### 不该进 skill 的内容 3：demo 页面临时工具逻辑

demo 里常有一些页面用的小辅助：

-   调试 GUI
-   本地缓存快照
-   临时按钮面板
-   只为演示方便而写的联动逻辑

这些内容可以当“模式说明”，但不能被写成引擎正式 API。

## 4. skill 文档应该写什么，不该写什么

一份可用的 skill，建议至少包含下面几个部分。

## 4.1 Quick Start

给出最小可用例子。

要求：

-   用公开 API
-   用当前版本有效入口
-   尽量让读者一眼看懂“应该从哪里下手”

## 4.2 Core Concepts / Architecture Positioning

这是 skill 的骨架。

这里要交代清楚：

-   真实入口是什么
-   对象边界是什么
-   生命周期在哪里
-   常见误解是什么

如果这部分不清楚，后面的示例再多也会误导。

## 4.3 Common Patterns / Demo-backed Patterns

这部分不是堆代码，而是把 demo 中已经验证过的组织方式提炼成模式。

例如：

-   “先 `load`，再加控件”
-   “先加载 `SceneTileset`，再在 `sourceLoaded` 后做样式控制”
-   “先批量实例化，再用模型调整控件修正姿态”

## 4.4 Failure Cases / Do Not Do This

这是 skill 和普通文档最容易拉开差距的地方。

一定要写：

-   常见错误路径
-   为什么错
-   正确替代方式是什么

因为很多工程问题不是“不会用”，而是“用错入口”。

## 4.5 Performance Tips

MineMap 里很多功能都和渲染成本有关。

所以适合补充：

-   什么时候该关拾取
-   什么时候不该常驻某个调试控件
-   什么时候不要高频 `setStyle()`
-   什么时候要限制并发 3D 组件数量

## 4.6 See Also

每个 skill 都不应该是孤岛。

需要把上下游 skill 串起来。

例如：

-   控件 skill 要连到 fundamentals 和 events
-   scene component skill 要连到 transforms、materials、performance
-   style skill 要连到 layer skill 和 style/data skill

## 5. 经验分享：如何保证“正确、准确、好用”

这部分是整个技能库整理中最重要的经验。

## 5.1 正确，比全面更重要

宁可少写，也不要把不确定的内容写成肯定结论。

如果一个能力还没被源码和 demo 同时验证，就先不写死。

原因是：

-   编码助手一旦吸收错误路径，会持续放大错误
-   错误的“推荐用法”比没有文档更危险

## 5.2 准确，来自边界判断，不只是参数抄写

准确不等于参数抄全。

真正的准确，是写清楚：

-   这个对象是什么
-   它不是什么
-   它该在什么时候用
-   它不该被拿去解决什么问题

例如 `EarthRotationControl`：

-   它公开导出
-   但它不是标准 `IControl`
-   所以不能误写成 `map.addControl(new minemap.EarthRotationControl(...))`

这类“边界准确”比参数表更重要。

## 5.3 好用，来自“能直接写代码”

一份好用的 skill 必须帮助读者快速下手。

所以写法上应偏向：

-   真实入口
-   最小例子
-   常见组合方式
-   常见错误避坑

而不是长篇概念讲解。

## 5.4 优先沉淀“决策信息”

真正能帮助编码助手和开发者的，不只是 API 名称，而是决策信息：

-   该选哪个入口
-   该拆还是该合
-   该何时初始化
-   该何时等待加载完成
-   该避免哪些旧路径

如果文档能帮助做决策，它就有长期价值。

## 6. 拆 skill 还是并 skill：一个实用判断法

这是维护技能库时经常要做的决策。

可以用下面四个问题来判断。

### 问题 1：是否共享同一核心入口？

如果共享同一入口，比如都围绕 `setStyle()` 和 style v8 结构，可以放在同一 skill。

### 问题 2：是否共享同一生命周期？

如果都需要在相同阶段执行，比如都依赖 `load` 后再挂载，适合放一起。

### 问题 3：失败点是否相似？

如果失败点完全不同，说明应拆开。

例如：

-   layer 管理的失败点是顺序、过滤、source 绑定、zoom 范围
-   style 切换的失败点是 diff、keepUserInfo、style.load 重建

这两类应拆开。

### 问题 4：读者是否会以同一种问题来搜索它？

如果用户通常会问成同一类问题，就适合放一起。

例如：

-   “怎么加控件、怎么移除控件、怎么写自定义 widget”
-   这就是明显的同一类问题

所以 `minemap-widget-and-controls` 是合理独立专题。

## 7. 推荐的实际工作流

如果以后还要继续扩展这套技能库，可以按下面的顺序操作。

## 7.1 选题来源

选题通常来自四个地方：

1. `source/index.js` 的公开导出
2. `Map` 的大入口方法
3. `demo/src/config.js` 的 demo 分类
4. 用户高频提问主题

这四类交叉之后，基本就能知道哪些专题最值得补。

## 7.2 调研顺序

建议固定顺序：

1. 找公开入口
2. 找实现文件
3. 找文档注释
4. 找 demo 页面
5. 找相邻 skill 是否已有部分覆盖

这样能避免重复写、错拆包、漏边界。

## 7.3 写作顺序

建议写作顺序也固定：

1. 标题与 description
2. Quick Start
3. Core Concepts
4. Common Patterns
5. Failure Cases
6. Performance Tips
7. See Also

先搭骨架，再填内容，会比从头一口气写到底更稳定。

## 7.4 收尾动作

新增 skill 后，至少要做三件事：

1. 更新 `minemap-skills/README.md` 索引
2. 给相关旧 skill 补 `See Also`
3. 检查 Markdown 文件是否有格式或语义错误

这一步很重要，否则 skill 会变成“孤立文件”。

下面给一个适合继续扩展时使用的简化模板。

```markdown

[最小可用示例]

-   真实入口
-   生命周期

-   原因
-   替代方案

## Performance Tips

-   渲染成本
-   常驻风险
-   高频调用风险

## See Also

-   相邻主题 skill
```

## 9. 一份 skill 的验收清单

在把一份新 skill 放进库之前，建议过一遍下面这个清单。

### 来源是否可靠

-   是否核对过 `source/`？
-   是否核对过 `docs/src-cn/`？
-   是否至少有一个 demo 支撑主要流程？

### 边界是否准确

-   是否写清对象是什么、不是什么？
-   是否区分了正式 API 和 demo 辅助逻辑？
-   是否避免了私有字段与旧路径扩散？

### 是否真的好用

-   是否有最小可用示例？
-   是否有常见失败案例？
-   是否有推荐组合方式？

### 是否便于维护

-   是否命名合理？
-   是否和相邻 skill 重复过多？
-   是否补了 README 索引和互链？

如果这四组问题都能回答“是”，这份 skill 一般就已经具备可入库质量。

---

## 10. 最后的建议

如果要长期维护 MineMap skill 库，最重要的不是写得快，而是坚持同一套方法。

建议固定以下原则：

1. 源码优先，demo 验证，文档补语义
2. 只推荐公开 API
3. 不继续扩散废弃路径
4. 不把 demo 临时写法写成正式规范
5. skill 优先服务“能直接写代码”，而不是服务“概念解释完整”

这套方法看起来比单纯写文档慢一些，但它的优势是：

-   结论稳定
-   复用价值高
-   更适合智能编码助手
-   也更适合团队长期沉淀

如果后续要继续扩 skill，直接按这份教程执行即可。
